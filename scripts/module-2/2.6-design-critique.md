# Lesson 2.6: Design Critique with AI

**ระยะเวลา:** 30 นาที | **Level:** 🚀 Intermediate

---

## 🎬 HOOK

> "ทุกคนต้องการ Feedback แต่ไม่มีใครว่าง"
> 
> Senior designer ยุ่ง... Peers ก็ยุ่ง... Design review ถูกเลื่อนไปเรื่อยๆ
> 
> **AI เป็น Design Critic ที่พร้อมให้ Feedback ตลอด 24/7** — ไม่มีอารมณ์ ไม่มี bias ส่วนตัว และให้ feedback ได้ทันที

---

## 🎯 LEARNING OBJECTIVES

เมื่อจบบทนี้ คุณจะสามารถ:

1. ✅ ใช้ AI ให้ Design Feedback ที่มีคุณภาพ
2. ✅ สร้าง Critique Prompts ที่ได้ feedback ที่ใช้ได้จริง
3. ✅ รู้ข้อจำกัดของ AI Critique และใช้อย่างเหมาะสม
4. ✅ Combine AI feedback กับ Human review

## 📚 CONTENT

### Part 1: AI Critique Framework (10 นาที)

#### ทำไม AI ถึงช่วย Critique ได้ดี?

| 👤 Human Critique | 🤖 AI Critique |
|-------------------|----------------|
| มี context ดีกว่า | Consistent criteria |
| เข้าใจ nuances | ไม่มี personal bias |
| รู้ brand history | Available 24/7 |
| อาจ miss details เพราะคุ้นเคย | Fresh eyes ทุกครั้ง |
| ใช้เวลา schedule | Instant feedback |

**Best approach:** ใช้ทั้งคู่! AI = First pass, Human = Final review

---

#### Master Prompt: Comprehensive Design Critique

```
# Role
คุณคือ Senior UX Designer ที่มีประสบการณ์ 10+ ปี
ให้ Feedback ตรงไปตรงมา แต่ Constructive

# Context
- Product: [app type / website type]
- Target users: [demographics + needs]
- Business goals: [what we want to achieve]
- Design stage: [wireframe / mockup / final]
- Constraints: [time, tech, brand guidelines]

# Design to Review
[Describe the design in detail OR mention that you're uploading an image]

Key elements:
- Screen/Page: [name]
- Main purpose: [what user should do]
- Key components: [list main elements]

# Critique Framework
วิเคราะห์ใน 6 dimensions:

1. **Visual Hierarchy**
   - Eye flow เป็นอย่างไร?
   - สิ่งสำคัญเด่นพอไหม?
   - มี Visual noise ไหม?

2. **Usability**
   - Users เข้าใจง่ายไหม?
   - Actions ชัดเจนไหม?
   - มี Friction points ไหม?

3. **Accessibility**
   - Color contrast เพียงพอไหม?
   - Text readable ไหม?
   - Touch targets ใหญ่พอไหม?

4. **Consistency**
   - ตรงกับ Design system ไหม?
   - มี Pattern ที่ขัดกันไหม?
   - Spacing consistent ไหม?

5. **Emotional Impact**
   - ให้ความรู้สึกอย่างไร?
   - ตรงกับ Brand ไหม?
   - Trustworthy ไหม?

6. **Goal Alignment**
   - ช่วยให้ User ทำ goal ได้ไหม?
   - มี Distractions ไหม?
   - CTA ชัดเจนไหม?

# Output Format
สำหรับแต่ละ Issue ที่พบ:

🔴/🟡/🟢 [Severity: Critical/Medium/Low]
**Issue:** [อธิบายปัญหา]
**Why it matters:** [ทำไมถึงเป็นปัญหา]
**Suggestion:** [แนะนำวิธีแก้]
**Example:** [ตัวอย่างถ้ามี]

# Prioritization
จัดอันดับ Issues ตาม Impact × Effort
```

### Part 2: Critique Types & Prompts (12 นาที)

#### Type 1: Quick Review (5 นาที)

สำหรับเมื่อต้องการ feedback เร็วๆ

```
ดู Design นี้แล้วบอก:

✅ 3 สิ่งที่ทำได้ดี (Strengths)
⚠️ 3 สิ่งที่ควรปรับปรุง (Improvements)
🎯 1 สิ่งที่ควรแก้ก่อน (Priority #1)

ให้ Feedback แบบ Actionable — บอกว่าทำอะไร ไม่ใช่แค่บอกปัญหา

Design: [description]
```

**Example Output:**
```
✅ Strengths:
1. Visual hierarchy ชัดเจน - Title เด่น, CTA สีเด่น
2. White space ดี - ไม่รู้สึก cluttered
3. Copy กระชับ - เข้าใจ value prop ใน 3 วินาที

⚠️ Improvements:
1. Color contrast ของ secondary text อาจไม่ผ่าน WCAG
2. Touch targets เล็กไป สำหรับ mobile
3. ไม่มี visual feedback ว่า form valid หรือยัง

🎯 Priority #1:
แก้ Color contrast ก่อน — เพิ่ม contrast ratio จาก ~3:1 เป็น 4.5:1
```

#### Type 2: Deep Dive Review (15-20 นาที)

สำหรับ design ที่สำคัญ หรือก่อน handoff

```
ทำ Deep Dive Design Review:

## Design Context
- Screen: [name]
- User journey: [where in the flow]
- Key action: [what user should do]

## Analysis Sections

### 1. First Impression Test
ถ้าเห็นครั้งแรก 5 วินาที:
- รู้ว่าเป็นอะไรไหม?
- รู้ว่าต้องทำอะไรไหม?
- อยากใช้ต่อไหม?

### 2. Visual Hierarchy Audit
- F-pattern / Z-pattern analysis
- สิ่งที่เด่นที่สุด 1, 2, 3 คืออะไร
- ตรงกับ priority ที่ต้องการไหม

### 3. Interaction Analysis
- Clickable elements ชัดเจนไหม
- Hover/Active states
- Loading/Empty states

### 4. Accessibility Checklist
- [ ] Color contrast ≥ 4.5:1
- [ ] Font size ≥ 16px body
- [ ] Touch targets ≥ 44px
- [ ] Alt text concepts
- [ ] Focus states

### 5. Edge Cases
- Empty state
- Error state
- Loading state
- Too much content
- Too little content

## Output
Severity-ranked issues พร้อม solutions
```

#### Type 3: Comparison Review

เมื่อต้องเลือกระหว่าง options

```
เปรียบเทียบ 2 Design Options:

## Option A
[describe or reference]

## Option B
[describe or reference]

## Evaluation Criteria
1. Usability (weight: 30%)
2. Visual appeal (weight: 25%)
3. Brand alignment (weight: 20%)
4. Development effort (weight: 15%)
5. Scalability (weight: 10%)

## Analysis
สำหรับแต่ละ criteria:
- Option A: score + reasoning
- Option B: score + reasoning

## Recommendation
- Overall winner: [A/B/Hybrid]
- For what use case: [explain]
- What to take from each option
```

### Part 3: Advanced Techniques (8 นาที)

#### Technique 1: Persona-Based Critique

```
Review design นี้จากมุมมองของ 3 personas:

## Persona 1: Tech-savvy millennial
- คาดหวังอะไร?
- จะ frustrated ตรงไหน?
- จะ delight ตรงไหน?

## Persona 2: Busy professional (40+)
- คาดหวังอะไร?
- จะ frustrated ตรงไหน?
- จะ delight ตรงไหน?

## Persona 3: First-time user
- คาดหวังอะไร?
- จะ confused ตรงไหน?
- ต้องการ guidance ตรงไหน?
```

---

#### Technique 2: Competitive Lens Critique

```
Review design นี้เทียบกับ Competitors:

## Best-in-class comparison
- [Competitor A] ทำตรงนี้ดีกว่าอย่างไร?
- [Competitor B] ทำตรงนี้ดีกว่าอย่างไร?

## Differentiation check
- Design นี้ต่างจาก competitors อย่างไร?
- ความต่างนั้นเป็น strength หรือ weakness?

## Industry standards
- มี patterns ที่ users คาดหวังไหม?
- มีอะไรที่ขัดกับ conventions ไหม?
```

---

#### Technique 3: Pre-mortem Critique

```
สมมติว่า Design นี้ Launch แล้ว FAIL:

## What went wrong?
- Users complain เรื่องอะไร?
- Metrics ตัวไหนไม่ดี?
- Support tickets เรื่องอะไร?

## Why did it fail?
- Design decisions ไหนที่เป็นสาเหตุ?
- อะไรที่เราควรเห็นแต่ miss?

## Prevention
- แก้อะไรก่อน launch?
- Test อะไรเพิ่ม?
```

## ⚠️ LIMITATIONS & BEST PRACTICES

### AI Critique Limitations

| ⚠️ Limitation | 💡 Workaround |
|--------------|---------------|
| ไม่เห็น actual interaction | Describe interactions in prompt |
| ไม่รู้ brand context ทั้งหมด | Include brand guidelines |
| อาจ miss cultural nuances | Add local context |
| ไม่แทน real user testing | Use as complement, not replacement |
| May hallucinate issues | Verify critical feedback |

### Best Practices Checklist

✅ **DO:**
- Use AI as first pass before human review
- Include context (brand, users, goals)
- Ask for actionable suggestions
- Verify accessibility issues with tools
- Combine with real user feedback

❌ **DON'T:**
- Replace user testing entirely
- Accept all suggestions blindly
- Skip human design review
- Ignore context-specific feedback
- Over-rely on AI for final decisions
